# 렌더링
## SSG (Static Site Generation)
- 렌더링 시점
  - 빌드 시점에 HTML 생성 / 정적파일
- 특징
  - 서버부담이 적음
  - 콘텐츠가 변경되지 않는 경우 적합
  - CDN을 통해 빠르게 제공가능
  - 데이터는 getStaticProps를 통해 가져옴
- 예시
  - 블로그 포스트, 마케팅 페이지, FAQ 정적 문서 등
- 장점
  - 성능이 뛰어나고 최초 로딩속도가 빠름
  - 서버 부하가 적고 스케일링에 유리
- 단점
  - 빌드 시 모든 HTML이 생성되므로 데이터가 자주 변경되거나 사용자 맞춤형 콘텐츠가 필요한 페이지에 부적합

## SSR (Server-Side Rendering)
- 렌더링 시점
  - 요청 시점에 HTML을 생성합니다.
  - 클라이언트의 요청이 들어올 때마다 서버에서 데이터를 가져와 HTML을 생성하고 응답합니다.
- 주요 특징
  - 매번 최신 데이터를 반영합니다.
  - 데이터가 자주 변경되거나 사용자별 맞춤 콘텐츠가 필요한 경우 적합합니다.
  - 데이터는 getServerSideProps를 통해 가져옵니다.
- 사용 예시
  - 대시보드, 실시간 데이터가 필요한 페이지, 사용자 맞춤형 콘텐츠 제공.
- 장점
  - 최신 데이터를 항상 제공합니다.
  - 사용자 맞춤형 페이지에 적합합니다.
- 단점
  - 요청마다 서버에서 HTML을 생성하므로 서버 부하가 클 수 있습니다.
  - 최초 로딩 속도가 느릴 수 있습니다.

## CSR (Client-Side Rendering)
렌더링 시점
1. 초기 HTML
  - 서버는 빈 HTML 틀(Skeleton)을 브라우저로 보냅니다.
  - 초기에는 <div id="root"></div>와 같은 최소한의 구조만 제공됩니다.
2. JavaScript 실행
  - 브라우저가 HTML을 다운로드한 후, JavaScript 번들을 로드하고 실행합니다.
  - React가 Virtual DOM을 생성하고, 데이터 요청(fetch)을 통해 필요한 데이터를 가져옵니다.
3. 데이터 바인딩
  - 데이터를 받아와 DOM에 결합하여 화면을 렌더링합니다.
**주요 특징**
- 모든 렌더링이 클라이언트(브라우저)에서 발생합니다.
- 사용자는 JavaScript가 실행되기 전까지 화면에 아무것도 볼 수 없거나 로딩 화면만 볼 수 있습니다.

- 장점
  - 유연성: 브라우저 환경에서 데이터를 동적으로 가져와 사용할 수 있습니다.
  - 빠른 초기 로딩: 초기에는 빈 HTML만 제공되므로 서버 부담이 적습니다.
  - 인터랙티브한 애플리케이션: React 상태 관리나 클라이언트에서 동적으로 데이터 변화를 처리하는 경우 적합합니다.
- 단점
  - SEO 문제: 초기 HTML에는 콘텐츠가 없으므로 검색 엔진 최적화(SEO)가 어렵습니다.
(Next.js는 기본적으로 SEO를 위해 SSR과 SSG를 권장합니다.)
  - 느린 첫 화면: JavaScript가 로드되고 실행될 때까지 사용자는 빈 화면이나 로딩 화면을 보게 됩니다.
   - JavaScript 의존성: 브라우저에서 JavaScript가 비활성화되면 애플리케이션이 작동하지 않을 수 있습니다.


## CSR vs SSR/SSG
|특징|CSR|	SSR/SSG|
|---|---|---|
|렌더링 위치|	클라이언트(브라우저)에서|	서버에서
|SEO|	약함|	강함|
|초기 로딩 속도|	느림|	빠름|
|데이터 처리 방식|	브라우저에서 동적 데이터 요청|	서버에서 데이터를 준비해 응답|
|적합한 용도|	실시간 데이터 업데이트가 많은 SPA|	SEO가 중요한 페이지, 정적/동적 콘텐츠 제공|
